# Quickstart & Validation Guide: Legal AI Agent Harness

This guide proves the feature works end-to-end. It references [contracts/](./contracts/) and [data-model.md](./data-model.md) rather than duplicating them. Implementation details live in `tasks.md` (generated by `/speckit-tasks`).

## Prerequisites

- Elixir ~> 1.15 / OTP, PostgreSQL running locally with the **pgvector** extension available.
- An OpenAI API key.
- A `.env` file at the repo root (git-ignored). Copy from `.env.example`:

```
DATABASE_URL=ecto://postgres:postgres@localhost/api_harness_dev
OPENAI_API_KEY=sk-...
JWT_SECRET=<random 64+ char secret>
SECRET_KEY_BASE=<mix phx.gen.secret>
```

## Setup

```bash
mix setup            # deps.get + ecto.create + migrate (enables pgvector) + seed
iex -S mix phx.server
```

## Scenario 1 — Create a user via the REPL (US1 / FR-001)

In the IEx session:

```elixir
{:ok, user} =
  ApiHarness.Accounts.create_user(%{
    name: "Eduardo",
    email: "eduardo@example.com",
    password: "s3cr3tpass"
  })

ApiHarness.Accounts.list_users()          # => [%User{...}]
```

**Expected**: user persisted; duplicate email returns `{:error, changeset}` (FR-003). Verifies SC-007 (CRUD fully from REPL, no HTTP).

## Scenario 2 — Authenticate (FR-000)

```bash
curl -s localhost:4000/api/login \
  -H 'content-type: application/json' \
  -d '{"email":"eduardo@example.com","password":"s3cr3tpass"}'
```

**Expected**: `200` with `{"token": "...", "user": {...}}` per [contracts/auth.md](./contracts/auth.md). Export `TOKEN=<token>`.

## Scenario 3 — Create and list threads (US2 / FR-004, FR-005)

```bash
curl -s localhost:4000/api/chats -H "authorization: Bearer $TOKEN" \
  -H 'content-type: application/json' -d '{"title":"Ação trabalhista"}'

curl -s localhost:4000/api/chats -H "authorization: Bearer $TOKEN"
```

**Expected**: `201` then a list containing the thread. A fresh session memory exists for the new thread (DB check on `session_memories`). Verifies SC-008 isolation when repeated for a second thread.

## Scenario 4 — Send a message, get an AI response (US3 / FR-008–FR-013)

```bash
curl -s localhost:4000/api/chats/42/messages -H "authorization: Bearer $TOKEN" \
  -H 'content-type: application/json' \
  -d '{"content":"Qual o prazo prescricional de uma ação trabalhista?"}'
```

**Expected**: `200` with an `assistant` message (per [contracts/messages.md](./contracts/messages.md)). Both the user message and AI response are persisted under the thread (DB check on `messages`). Verifies SC-001. The response returns before memory writes occur (SC-002).

**Negative checks**:
- Empty `content` → `400`.
- Unknown/foreign `chat_id` → `404`.
- Simulated OpenAI outage (point the provider at an unreachable base URL or use the test stub returning an error) → `502/503`, no retry, no fallback (FR-013-A).

## Scenario 5 — Memory is updated asynchronously (US4, US5 / FR-014–FR-024)

After Scenario 4 completes, give the pipeline a moment (eventual consistency), then inspect the DB:

```elixir
ApiHarness.Memory.get_session_memory(chat_id)      # => %{state: %{...extracted facts...}}
ApiHarness.Memory.list_persistent_memories(user_id) # => reconciled records
Repo.aggregate(MemoryContextUpdate, :count)         # => audit rows for create/update/merge/discard
```

**Expected**:
- Session memory `state` reflects facts from the conversation (SC-004).
- Sending a follow-up that restates known knowledge does **not** grow the persistent memory row count — it updates/merges (SC-005, FR-017).
- Retrieval for a new, unrelated question pulls only relevant memories, excluding off-topic categories (SC-006).

> In tests, never use `Process.sleep/1`. Synchronize on the pipeline worker with `Process.monitor/1` + `assert_receive {:DOWN, ...}` or `:sys.get_state/1` (constitution Test Discipline), and start processes with `start_supervised!/1`.

## Scenario 6 — Concurrency (US5 / FR-024, SC-003)

Fire several `POST /messages` for different users concurrently (e.g., a small script or `Task.async_stream`). **Expected**: each response returns independently; each user's memory pipeline runs in its own supervised worker with no cross-blocking and no lost updates.
